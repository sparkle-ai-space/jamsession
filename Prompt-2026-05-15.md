# WIP: Dispatcher Architecture

**Branch**: `conductor-like-arch` | **Design doc**: `md/design/daemon_actor.md`

## Core Idea

The daemon has three kinds of tasks:

1. **Dispatcher** — single task, owns all mutable state, processes messages sequentially. All protocol logic lives here.
2. **Client pipes** — one per client connection. Dumb bidirectional forwarders. No protocol knowledge.
3. **Agent pipes** — one per agent process. Same shape as client pipes.

No `Arc<Mutex<...>>`. No dynamic handlers on the client side. Agent pipes use a single dynamic handler installed post-handshake. The dispatcher is an actor — message passing only.

## Client Pipe

A client pipe is spawned per accepted Unix socket connection. It has:

- An `on_receive_dispatch` handler that forwards *every* dispatch (requests, notifications, everything) to the dispatcher.
- A spawned task that reads from an `rx` channel and sends outgoing dispatches back to the ACP connection.

```rust
// Pseudocode for the client pipe task
async fn client_pipe(stream: UnixStream, dispatcher_tx: Sender<DispatcherMessage>) {
    let (outgoing_tx, outgoing_rx) = mpsc::unbounded_channel::<Dispatch>();
    let client_id = generate_id();

    // Register with dispatcher
    dispatcher_tx.send(DispatcherMessage::ClientRegistered {
        client_id,
        outgoing_tx,
    });

    Agent
        .builder()
        .name("jamsession-daemon")
        .on_receive_dispatch(async |dispatch, _cx| {
            dispatcher_tx.send(DispatcherMessage::FromClient { client_id, dispatch });
            Ok(Handled::Yes)
        })
        .with_spawned(|cx| async move {
            // Read from outgoing_rx, send each dispatch back to the client
            while let Some(dispatch) = outgoing_rx.recv().await {
                cx.send_proxied_message(dispatch)?;
            }
            Ok(())
        })
        .connect_to(transport)
        .await;

    // Connection closed
    dispatcher_tx.send(DispatcherMessage::ClientDisconnected { client_id });
}
```

## Agent Pipe

Same shape as client pipes. Spawned by the dispatcher (via `tokio::spawn`) when it needs an agent — either for a capabilities probe or for a session.

Takes an `Option<SessionRequest>`:
- `None` → capabilities probe. Initialize the agent, report capabilities back to the dispatcher, then disconnect.
- `Some(SessionRequest::New { cwd, mcp_servers })` → full session. Initialize, validate capabilities against asserted baseline, send session/new, stay alive as the session's agent.
- `Some(SessionRequest::Load { session_id, cwd, mcp_servers })` → reconnect. Initialize, validate capabilities, send session/load, stay alive.

When `session_request` is `Some(...)`, the caller also passes in the expected capabilities (the baseline established during the first `Initialize`). The agent pipe validates the agent's response against this baseline — if the agent provides fewer capabilities than expected, it errors out (sends `AgentSpawnFailed` and disconnects).

This keeps the dispatcher loop non-blocking — all multi-step async work (connect, initialize, session setup) happens in the agent pipe task.

```rust
enum SessionRequest {
    New { cwd: PathBuf, mcp_servers: Vec<McpServer> },
    Load { session_id: String, cwd: PathBuf, mcp_servers: Vec<McpServer> },
}

enum AgentSpawnMode {
    /// Capabilities probe — initialize, report capabilities, disconnect.
    Probe,
    /// Full session — initialize, validate capabilities, establish session, stay alive.
    Session {
        request: SessionRequest,
        expected_capabilities: ExpectedCapabilities,
    },
}

struct AgentSpawnRequest {
    client_id: ClientId,
    request_id: RequestId,
    mode: AgentSpawnMode,
}

// Pseudocode for the agent pipe task
async fn agent_pipe(
    transport: DynConnectTo<Client>,
    dispatcher_tx: Sender<DispatcherMessage>,
    spawn_request: AgentSpawnRequest,
) {
    let (outgoing_tx, outgoing_rx) = mpsc::unbounded_channel::<Dispatch>();
    let agent_id = generate_id();

    let result = Client
        .builder()
        .name("jamsession-daemon-agent")
        .connect_with(transport, async |cx| {
            // --- Handshake phase (no forwarding handler yet) ---
            // Responses route normally via internal oneshot correlation.
            let init_resp = cx.send_request(InitializeRequest::new(...)).block_task().await?;

            match spawn_request.mode {
                AgentSpawnMode::Probe => {
                    // Capabilities probe only — report and disconnect
                    dispatcher_tx.send(DispatcherMessage::AgentCapabilities {
                        capabilities: init_resp,
                    });
                    return Ok(());
                }
                AgentSpawnMode::Session { request, expected_capabilities } => {
                    // Validate capabilities against baseline
                    if !expected_capabilities.satisfied_by(&init_resp) {
                        dispatcher_tx.send(DispatcherMessage::AgentSpawnFailed {
                            client_id: spawn_request.client_id,
                            error: Error::CapabilityMismatch,
                        });
                        return Ok(());  // disconnect
                    }

                    match request {
                        SessionRequest::New { cwd, mcp_servers } => {
                            let resp = cx.send_request(NewSessionRequest::new(cwd)...).block_task().await?;

                            // Send guidelines prompt (if configured)
                            if send_guidelines {
                                cx.send_request(PromptRequest::new(
                                    resp.session_id, vec![ContentBlock::Text(GUIDELINES)]
                                )).block_task().await?;
                            }

                            cx.add_dynamic_handler(AgentForwarder { agent_id, dispatcher_tx })?
                              .run_indefinitely();
                            dispatcher_tx.send(DispatcherMessage::AgentReady {
                                agent_id,
                                outgoing_tx,
                                session_id: resp.session_id,
                                client_id: spawn_request.client_id,
                            });
                        }
                        SessionRequest::Load { session_id, cwd, mcp_servers } => {
                            // Install a temporary capture handler to collect notifications
                            // emitted during session/load (the agent replays history as
                            // notifications BEFORE sending the LoadSessionResponse).
                            let replay_buffer = Arc::new(Mutex::new(Vec::new()));
                            cx.add_dynamic_handler(ReplayCapture { buffer: replay_buffer.clone() })?
                              .run_indefinitely();

                            cx.send_request(LoadSessionRequest::new(...)).block_task().await?;
                            // Response arrived = all replay notifications are captured.

                            // Switch to forwarding handler for ongoing traffic.
                            cx.add_dynamic_handler(AgentForwarder { agent_id, dispatcher_tx })?
                              .run_indefinitely();

                            let replay = replay_buffer.lock().unwrap().clone();
                            dispatcher_tx.send(DispatcherMessage::AgentReady {
                                agent_id,
                                outgoing_tx,
                                session_id,
                                client_id: spawn_request.client_id,
                                replay_notifications: replay,
                            });
                        }
                    }
                }
            }

            // Keep connection alive by pumping outgoing dispatches.
            // When the dispatcher drops outgoing_tx, recv() returns None and we exit naturally.
            while let Some(dispatch) = outgoing_rx.recv().await {
                cx.send_proxied_message(dispatch)?;
            }
            Ok(())
        })
        .await;

    // If we get here, agent connection closed
    dispatcher_tx.send(DispatcherMessage::AgentDisconnected { agent_id });
}
```

### AgentForwarder

A trivial dynamic handler that claims every dispatch and forwards it to the dispatcher. Installed post-handshake so it doesn't interfere with `block_task()` during initialization.

```rust
struct AgentForwarder {
    agent_id: AgentId,
    dispatcher_tx: mpsc::UnboundedSender<DispatcherMessage>,
}

impl HandleDispatchFrom<agent_client_protocol::Agent> for AgentForwarder {
    async fn handle_dispatch_from(
        &mut self,
        message: Dispatch,
        _cx: ConnectionTo<agent_client_protocol::Agent>,
    ) -> agent_client_protocol::schema::Result<Handled<Dispatch>> {
        self.dispatcher_tx.send(DispatcherMessage::FromAgent {
            agent_id: self.agent_id,
            dispatch: message,
        }).map_err(agent_client_protocol::Error::internal_error)?;
        Ok(Handled::Yes)
    }

    fn describe_chain(&self) -> impl std::fmt::Debug {
        "AgentForwarder"
    }
}
```

That's it — no filtering, no deserialization, no state. Requests, notifications, and responses all go to the dispatcher as opaque `Dispatch` values. The dispatcher decides what to do with them.

### Dispatch type and response routing

`Dispatch` has three variants: `Request`, `Notification`, and `Response`. When a response arrives on the wire, it flows through the handler chain — if a handler claims it (`Handled::Yes`), it's intercepted; otherwise it falls through to the internal oneshot-based correlation mechanism (so `block_task().await` works).

This is why we install the forwarding handler **after** the handshake: during handshake, responses flow to `block_task()` normally. Once the handler is installed (claiming all dispatches), subsequent dispatches — including responses to requests sent *by the dispatcher through the pipe* — are forwarded to the dispatcher for routing.

## Task Spawner (`scope-tasks` crate)

A lightweight structured-concurrency primitive: a scoped concurrent task runner. Designed as a standalone crate (`src/scope-tasks/`).

### Concept

A `Scope` runs a "main" future (the foreground) alongside a set of dynamically-spawned concurrent tasks (the background). Tasks are submitted via a `TaskSpawner` handle. When the foreground completes, all in-flight tasks are cancelled (dropped).

Unlike `tokio::spawn`, tasks are **scoped** — they are guaranteed to complete (or be dropped) before `scope()` returns. Because of this guarantee, spawned tasks do **not** require `'static` — they can borrow from the scope's caller. This is the same insight as `moro` / `crossbeam::scope`: the scope function doesn't return until all tasks are done, so borrows into tasks are sound.

The API uses a lifetime-bound closure (like `std::thread::scope`) to enforce this:

```rust
scope(|spawner| async move {
    // `spawner.spawn(...)` accepts futures that borrow from this stack frame
    ...
})
```

### API

```rust
// src/scope-tasks/src/lib.rs

use std::pin::Pin;
use std::future::Future;
use tokio::sync::mpsc;

/// Handle for submitting tasks into a scope. Clone-able, Send-able.
/// When all clones are dropped, the background runner has no new work and will exit
/// once in-flight tasks complete (or be cancelled if the scope foreground finishes first).
///
/// The lifetime `'scope` ties spawned futures to the enclosing scope —
/// they can borrow from the caller's stack frame.
///
/// Generic over error type `E` so callers choose their own error domain.
#[derive(Clone)]
pub struct TaskSpawner<'scope, E> {
    tx: mpsc::UnboundedSender<Pin<Box<dyn Future<Output = Result<(), E>> + Send + 'scope>>>,
}

/// Trait for constructing an error when `spawn` is called on a closed scope.
/// Analogous to how `Default` provides a value from nothing.
pub trait SpawnError {
    fn scope_closed() -> Self;
}

impl<'scope, E: SpawnError> TaskSpawner<'scope, E> {
    /// Spawn a task to run concurrently in the scope's background.
    /// The future can borrow data that outlives the scope.
    ///
    /// Returns `Err` if the scope has already shut down (background runner exited).
    pub fn spawn(&self, fut: impl Future<Output = Result<(), E>> + Send + 'scope) -> Result<(), E> {
        self.tx.send(Box::pin(fut)).map_err(|_| E::scope_closed())
    }
}

/// Run `main_fn` as the foreground, with a concurrent task runner in the background.
///
/// `main_fn` receives a `TaskSpawner` it can use to kick off concurrent work.
/// Spawned tasks can borrow from the enclosing stack frame (they don't need `'static`).
///
/// When `main_fn` returns, in-flight background tasks are cancelled (dropped)
/// and `scope` returns the foreground's result.
///
/// If any background task returns `Err`, the error propagates and the scope exits.
pub async fn scope<'env, T, E>(
    main_fn: impl for<'scope> AsyncFnOnce(&'scope TaskSpawner<'scope, E>) -> Result<T, E> + 'env,
) -> Result<T, E> {
    let (tx, rx) = mpsc::unbounded_channel();
    let spawner = TaskSpawner { tx };

    run_until(
        run_tasks(rx),
        main_fn(&spawner),
    ).await
}

/// Background runner: receives tasks from the channel and runs them concurrently
/// in a FuturesUnordered. Exits when the channel is closed and all tasks complete
/// (or is cancelled by `run_until` when the foreground finishes first).
///
/// Uses the same pattern as `process_stream_concurrently` from agent-client-protocol:
/// race between "new item from stream" and "a running future completed".
async fn run_tasks<'scope, E>(
    rx: mpsc::UnboundedReceiver<Pin<Box<dyn Future<Output = Result<(), E>> + Send + 'scope>>>,
) -> Result<(), E> {
    use std::pin::pin;
    use futures::stream::{FusedStream, FuturesUnordered, StreamExt};
    use futures_concurrency::future::Race;

    enum Event<T> {
        NewTask(Option<T>),
        TaskCompleted(Option<Result<(), E>>),
    }

    let mut stream = pin!(UnboundedReceiverStream::new(rx).fuse());
    let mut tasks: FuturesUnordered<_> = FuturesUnordered::new();

    loop {
        // No running tasks — wait for one
        if tasks.is_empty() {
            match stream.next().await {
                Some(task) => { tasks.push(task); continue; }
                None => return Ok(()),
            }
        }

        // Stream exhausted — drain remaining tasks
        if stream.is_terminated() {
            while let Some(result) = tasks.next().await {
                result?;
            }
            return Ok(());
        }

        // Race: new task vs. a running task completing
        let event = (
            async { Event::NewTask(stream.next().await) },
            async { Event::TaskCompleted(tasks.next().await) },
        ).race().await;

        match event {
            Event::NewTask(Some(task)) => { tasks.push(task); }
            Event::TaskCompleted(Some(result)) => { result?; }
            Event::NewTask(None) | Event::TaskCompleted(None) => {}
        }
    }
}
```

**Note on soundness**: The `'scope` lifetime ensures spawned futures cannot outlive the scope. When the foreground returns, `run_until` drops the background (cancelling in-flight tasks). Dropping a future that borrows from the enclosing scope is safe — drop runs synchronously and the borrows end immediately. Unlike `std::thread::scope` (which needs unsafe because threads are opaque to the borrow checker and could still be *running*), async futures are cooperative — dropping cancels them with no further access to borrowed data. This should be expressible in safe Rust since the entire future tree is local (no `tokio::spawn`).

### Usage in the Dispatcher

```rust
scope(async |spawner| {
    let mut dispatcher = Dispatcher::new(..., spawner);
    dispatcher.run(msg_rx).await
}).await
```

Inside the dispatcher, spawning work:

```rust
// Handling session/new — spawn a task that captures the Responder.
// No 'static required — the task can borrow from the scope.
self.spawner.spawn(async {
    let transport = factory.create_transport(...)?;
    let (ready_tx, ready_rx) = oneshot::channel();
    tokio::spawn(agent_pipe(transport, dispatcher_tx.clone(), ready_tx, ...));

    match ready_rx.await {
        Ok(agent_info) => {
            responder.respond(NewSessionResponse::new(agent_info.session_id));
            dispatcher_tx.send(DispatcherMessage::AgentReady { ... });
        }
        Err(_) => {
            responder.respond_with_error(Error::internal_error());
        }
    }
    Ok(())
});
```

### Error Semantics

- A spawned task returning `Err` propagates up and kills the scope (and thus the dispatcher). This is intentional for unrecoverable errors (broken transport, corrupted state).
- For recoverable errors (agent won't start, invalid request), the task should respond to the client with an error and return `Ok(())`.
- Panics in spawned tasks: TBD — likely catch_unwind at the task boundary and convert to `Err`.

### Why a Separate Crate

- General-purpose abstraction: useful anywhere you want a sequential "main" loop that can kick off concurrent side-work with structured teardown.
- No dependency on ACP or jamsession-specific types.
- Small API surface, easy to test independently.
- Publishable as a standalone utility.

## Dispatcher

### Concurrency Model

The dispatcher runs inside a `scope`. It owns the `TaskSpawner` and uses it to kick off async work (agent spawns, capability probes) that captures transient state like `Responder`s. Results flow back via the dispatcher's message channel.

**Two kinds of state**:
- **Shared state** (in the dispatcher struct): `sessions`, `clients`, `agents`, `capabilities`. Persists across many messages. Accessed directly in the dispatch loop.
- **Transient state** (captured in spawned tasks): a `Responder` waiting for an agent to come up, a one-time timer, etc. Lives only until the task completes. No maps, no IDs — the closure captures what it needs.

This eliminates the `pending_spawns` map for most cases — the spawned task *is* the pending state.

### Dispatcher State

The dispatcher struct owns:

- `clients: HashMap<ClientId, ClientHandle>` — outgoing tx per client
- `agents: HashMap<AgentId, AgentHandle>` — outgoing tx per agent
- `sessions: HashMap<SessionId, Session>` — maps session to its agent, connected clients, lifecycle state, notification buffer, generation counter
- `state: DaemonState` — persistent state (session records)
- `capabilities: Option<CachedCapabilities>` — stored baseline (set once)
- `task_spawner: TaskSpawner` — for kicking off async work

### Message Enum

```rust
enum DispatcherMessage {
    // --- Pipe registration/teardown ---
    ClientRegistered { client_id: ClientId, outgoing_tx: Sender<Dispatch> },
    ClientDisconnected { client_id: ClientId },
    AgentReady { agent_id: AgentId, outgoing_tx: Sender<Dispatch>, session_id: String, client_id: ClientId, replay_notifications: Vec<serde_json::Value> },
    AgentCapabilities { capabilities: InitializeResponse },
    AgentSpawnFailed { client_id: ClientId, error: Error },
    AgentDisconnected { agent_id: AgentId },

    // --- Forwarded dispatches ---
    FromClient { client_id: ClientId, dispatch: Dispatch },
    FromAgent { agent_id: AgentId, dispatch: Dispatch },

    // --- Timers ---
    AgentQuiescent { session_id: String, generation: u64 },
    IdleTimeoutElapsed { session_id: String, generation: u64 },
    CwdHealthCheck,
}
```

Note: `AgentCapabilities` and `AgentSpawnFailed` no longer carry a `spawn_request` — the `Responder` is captured in the spawned task that handles the response. The dispatcher just needs to know which client to update on failure, and to store the capabilities result.

### Handling `FromClient`

The dispatcher deserializes the dispatch and decides what to do:

| Dispatch | Action |
|----------|--------|
| `Initialize` request | See Capabilities section below. |
| `ListSessions` request | Respond immediately via the `Responder` extracted from the dispatch. |
| `NewSessionRequest` | Spawn a task (via `task_spawner`) that captures the `Responder`, spawns an agent pipe, and waits for `AgentReady` — then responds. See flow below. |
| `LoadSessionRequest` | Similar — check if agent alive, reuse or spawn new. |
| `ResumeSessionRequest` | Similar to load. |
| Dispatch for active session | Route to agent via `agents[agent_id].tx`. |

### Session ↔ Client ↔ Agent Relationships

```rust
struct Session {
    session_id: SessionId,
    agent_id: AgentId,
    /// Clients connected to this session, ordered by connection time.
    /// Last element = most recently connected = "active" client.
    client_ids: Vec<ClientId>,
    /// Buffered notifications for session/load replay.
    buffer: Vec<serde_json::Value>,
    generation: u64,
    lifecycle_state: LifecycleState,
    /// Guards against infinite respawn loops on repeated agent crashes.
    respawn_attempted: bool,
    /// Persistent record (cwd, timestamps). Saved to disk on mutation.
    record: SessionRecord,
}
```

Reverse lookups:
- `client_to_session: HashMap<ClientId, SessionId>` — so `.otherwise()` routing from a client can find its session's agent without scanning.
- `agent_to_session: HashMap<AgentId, SessionId>` — so `FromAgent` can find the session.

**Routing rules:**
- **Agent → clients (notification)**: broadcast to ALL `session.client_ids` via each client's `outgoing_tx`.
- **Agent → client (request/response)**: send to the *most recently connected* client only (`session.client_ids.last()`).
- **Client → agent**: any client in the session can send; route to `session.agent_id`'s `outgoing_tx`.

In practice there's almost always one client per session. Multi-client is an edge case (e.g., two terminals attached to the same session), but the design handles it.

### Handling `FromAgent`

| Dispatch | Action |
|----------|--------|
| Notification | Buffer for replay. Broadcast to all connected clients via their `outgoing_tx`. |
| Request | Forward to the most recently connected client (active client). |
| Response | Forward to the most recently connected client (active client). |
| Any | Bump generation (invalidates quiescence timers). |

### Session/New Flow (with spawned tasks)

The dispatcher uses `MatchDispatch` (from `agent_client_protocol::util`) to pattern-match typed requests from raw dispatches. No manual method-string checking or deserialization:

```rust
// In the dispatcher's FromClient handler:
DispatcherMessage::FromClient { client_id, dispatch } => {
    MatchDispatch::new(dispatch)
        .if_request(async |req: InitializeRequest, responder| {
            self.handle_initialize(client_id, req, responder).await
        }).await
        .if_request(async |req: ListSessionsRequest, responder| {
            self.handle_list_sessions(req, responder)
        }).await
        .if_request(async |req: NewSessionRequest, responder| {
            self.handle_session_new(client_id, req, responder).await
        }).await
        .if_request(async |req: LoadSessionRequest, responder| {
            self.handle_session_load(client_id, req, responder).await
        }).await
        .if_request(async |req: ResumeSessionRequest, responder| {
            self.handle_session_resume(client_id, req, responder).await
        }).await
        .otherwise(async |dispatch| {
            // Steady-state: route dispatch to session's agent
            self.route_to_agent(client_id, dispatch)
        }).await?;
}
```

And the session/new handler spawns a task capturing the responder:

```rust
async fn handle_session_new(
    &self,
    client_id: ClientId,
    req: NewSessionRequest,
    responder: Responder<NewSessionResponse>,
) {
    let factory = self.factory.clone();
    let dispatcher_tx = self.dispatcher_tx.clone();

    // Spawn a task that captures the responder.
    // The task lives until the agent is ready (or fails).
    self.task_spawner.spawn(async move {
        let transport = factory.create_transport(...)?;
        let (ready_tx, ready_rx) = oneshot::channel();
        tokio::spawn(agent_pipe(transport, dispatcher_tx.clone(), ready_tx, ...));

        match ready_rx.await {
            Ok(agent_info) => {
                // Tell dispatcher about the new session (shared state update)
                dispatcher_tx.send(DispatcherMessage::AgentReady {
                    agent_id: agent_info.agent_id,
                    outgoing_tx: agent_info.outgoing_tx,
                    session_id: agent_info.session_id,
                    client_id,
                });
                // Respond to the client directly (transient state — responder captured here)
                responder.respond(NewSessionResponse::new(agent_info.session_id));
            }
            Err(_) => {
                responder.respond_with_error(Error::internal_error());
            }
        }
        Ok(())
    });
}
```

The `Responder` lives in the spawned task — no map lookup, no correlation ID. The task does its work and responds when done. The `AgentReady` message still goes to the dispatcher so it can record the ongoing session state.

### Capabilities (simplified)

- **First `Initialize`**: Dispatcher spawns a task that captures the `Responder`. That task spawns a temp agent pipe (`session_request: None`), awaits the capabilities result (via a oneshot from the agent pipe), stores the baseline in the dispatcher (via a `DispatcherMessage::AgentCapabilities`), and calls `responder.respond(...)`.
- **Later `Initialize`**: Compare incoming client capabilities against baseline. If fewer → abort via `responder.respond_with_error(...)`. Otherwise → respond from cache immediately via `responder.respond(...)`.
- **Agent spawns (session/new, session/load)**: Agent pipe runs initialize during its handshake. If capabilities mismatch the stored baseline, the agent pipe kills itself — the oneshot drops, the spawned task gets `Err`, and responds with an error to the client.

### Timers (unchanged)

Generation-counter pattern. Disconnect → bump generation → spawn sleep task → task sends `AgentQuiescent { generation }` → dispatcher checks if generation still matches.

## Session Lifecycle (end-to-end)

### New session

```
Client ──dispatch(NewSessionRequest)──→ ClientPipe ──FromClient──→ Dispatcher
                                                                      │
                                                          task_spawner.spawn(...)
                                                                      │
                                              ┌───────────────────────┴───────────────────┐
                                              │ Spawned task (captures Responder)          │
                                              │   tokio::spawn(agent_pipe)                 │
                                              │   awaits oneshot from agent pipe            │
                                              └───────────────────────┬───────────────────┘
                                                                      │
AgentPipe: connect → initialize → session/new ──oneshot──→ Spawned task
                                                                      │
                                              ┌───────────────────────┴───────────────────┐
                                              │ responder.respond(NewSessionResponse)      │
                                              │ dispatcher_tx.send(AgentReady { ... })     │
                                              └───────────────────────────────────────────┘
```

After setup, steady-state routing:
```
Client ──dispatch──→ ClientPipe ──FromClient──→ Dispatcher ──dispatch──→ AgentPipe ──→ Agent
Agent  ──dispatch──→ AgentPipe  ──FromAgent──→  Dispatcher ──dispatch──→ ClientPipe ──→ Client
```

### Client disconnect

```
ClientPipe detects close ──ClientDisconnected──→ Dispatcher
Dispatcher: clear client_id from session, bump generation, start quiescence timer
```

### Agent crash

```
AgentPipe detects close ──AgentDisconnected──→ Dispatcher
Dispatcher: mark session dead, clear buffer
```

### Load session (agent dead)

```
Client ──dispatch(LoadSessionRequest)──→ ... ──FromClient──→ Dispatcher
Dispatcher: session exists, agent dead → task_spawner.spawn(...)
  Spawned task: tokio::spawn(agent_pipe with session/load), await oneshot
  Agent pipe: connect → initialize → session/load ──oneshot──→ spawned task
  Spawned task: responder.respond(...), dispatcher_tx.send(AgentReady { ... })
Dispatcher receives AgentReady: rewire session, replay buffer to client
```

### Load session (agent alive)

```
Client ──dispatch(LoadSessionRequest)──→ ... ──FromClient──→ Dispatcher
Dispatcher: session exists, agent alive → replay buffer to new client, update session's client_id
responder.respond(LoadSessionResponse) immediately.
```

## Additional Concerns (from review)

### Lifecycle events

The test framework (`jamsession-test`) depends on `LifecycleEvent` emissions for synchronization (waiting for `Initialized`, `SessionCreated`, etc.). The dispatcher must emit these at the appropriate points:
- `Initialized` — after the accept loop starts.
- `ClientConnected` / `ClientDisconnected` — on `ClientRegistered` / `ClientDisconnected`.
- `SessionCreated` — when `AgentReady` is processed for a new session.
- `SessionLoaded` / `SessionResumed` — when load/resume completes.
- `AgentQuiescent` / `AgentKilledIdle` — on timer events.

The dispatcher holds an `Option<LifecycleEventSender>` (same as current actor).

### Guidelines delivery

The agent pipe sends guidelines (a `PromptRequest` with `include_str!("guidelines.md")`) after `session/new`, before installing the forwarder. This is gated on a `send_guidelines: bool` config passed into the spawn request. (Tests disable it.)

### CWD health check timer

A periodic task (spawned at daemon startup, not inside the scope) sends `DispatcherMessage::CwdHealthCheck` every 60 seconds. Same pattern as current code.

### `ReplayCapture` during session/load

The agent replays history as notifications before sending `LoadSessionResponse`. The agent pipe installs a `ReplayCapture` dynamic handler before sending `session/load`, captures all notifications into a buffer, then includes them in `AgentReady` (as `replay_notifications`). The dispatcher stores these in `session.buffer` and replays them to connected clients.

### `DaemonState::save()` is blocking I/O

`state.save()` does synchronous `fs::write` + `fs::rename`. This blocks the dispatcher's event loop briefly. Acceptable for now (saves are rare — only on session create/remove). If it becomes a problem, move to `tokio::task::spawn_blocking`.

### Race: client sends prompt before `AgentReady` is processed

After `responder.respond(NewSessionResponse)`, the client may immediately send a prompt. If the dispatcher hasn't processed the `AgentReady` message yet (it's still in the channel), the prompt arrives as `FromClient` but the `.otherwise()` routing can't find the session.

Mitigation: the spawned task sends `AgentReady` *before* calling `responder.respond(...)`. Since `AgentReady` and `FromClient { prompt }` both go through the same `dispatcher_tx` channel, and `AgentReady` is sent first, the dispatcher will process `AgentReady` before the prompt. This is guaranteed by channel ordering (unbounded mpsc is FIFO).

## Design Principles

**No mutexes.** All shared mutable state lives in the dispatcher. Spawned tasks communicate via messages.

**Transient state in closures, persistent state in the struct.** A `Responder` for an in-flight request is captured by the spawned task that will fulfill it. Session maps, agent handles, capability baselines live in the dispatcher struct.

**Pipes are dumb.** Client and agent pipes have zero protocol knowledge. They forward bytes (dispatches) and signal connect/disconnect. All intelligence is in the dispatcher.

**Structured teardown.** When the dispatcher exits, `run_until` drops the background (cancelling in-flight tasks). Agent pipes are `tokio::spawn`'d independently — their channel `tx`s close, they exit naturally.

## FAQ (non-issues and common concerns)

**Q: How does the dispatcher respond to client requests if the client pipe is just a dumb forwarder?**

`Dispatch::Request(req, responder)` — the `Responder` travels *with* the dispatch through the channel. The dispatcher extracts it and calls `responder.respond(...)`, which writes directly back through the original ACP connection's internal machinery. The client pipe's `outgoing_tx` is only used for unsolicited messages (forwarded agent notifications/responses). Responses to requests always go through `Responder`.

**Q: Doesn't returning `Handled::Yes` from `on_receive_dispatch` mean the request is "handled" — so we can't respond later?**

No. `Handled::Yes` means "I claimed this dispatch, don't pass it to the next handler in the chain." It says nothing about whether a response has been sent. The `Responder` can be called at any later time (from a spawned task, after an async operation, etc.). This is the standard ACP pattern.

**Q: Won't the client send a prompt before the dispatcher processes `AgentReady`?**

No. The spawned task sends `AgentReady` *before* `responder.respond(NewSessionResponse)`. Both messages go through the same FIFO channel (`dispatcher_tx`). The client can't send a prompt until it receives the response, and the response isn't sent until after `AgentReady` is in the queue. Ordering guaranteed.

**Q: Does `connect_with` keep the connection alive while the callback future is pending?**

Yes. `connect_to` (the "run forever" server mode) is literally implemented as `connect_with(transport, async |_cx| future::pending().await)`. The agent pipe parking in an outgoing-dispatch loop inside `connect_with` is explicitly supported.

**Q: Is `Responder` `Send + 'static`?**

Yes. No lifetime parameters. Can be freely captured in spawned tasks and moved across channels.

**Q: Does `send_proxied_message` work with untyped `Dispatch`?**

Yes. `Dispatch` defaults to `Dispatch<UntypedMessage, UntypedMessage>`, and `UntypedMessage` implements both `JsonRpcRequest` and `JsonRpcNotification`.

**Q: Can `add_dynamic_handler` coexist with the `connect_with` callback still running?**

Yes. The dynamic handler is added to the connection's handler chain. Incoming messages flow through it. The callback's future runs independently (in our case, it's the outgoing dispatch pump loop).

**Q: What about `run_until` not draining background tasks?**

Not a problem. When the dispatcher (foreground) exits, all in-flight scoped tasks are cancelled (dropped). This is fine — the daemon is shutting down. Agent pipes (`tokio::spawn`'d independently) detect their `outgoing_tx` closing and exit naturally. Dropping a future is safe even if it borrows non-`'static` data — drop is synchronous, no further access.

**Q: Do we need `unsafe` for the non-`'static` scope?**

Probably not. The entire future tree is local (no `tokio::spawn` for scoped tasks). The compiler can see that the channel contents don't escape. `FuturesUnordered` doesn't require `'static`. The HRTB signature might be tricky to express, but that's an ergonomics issue, not a soundness one.

**Q: What about the `ReplayCapture` using a `Mutex`?**

The `Mutex` in `ReplayCapture` is `std::sync::Mutex`, not `tokio::sync::Mutex`. It's only used within the agent pipe's `connect_with` callback (single task context — the lock is held briefly to push a notification). This is fine and matches the existing code's approach.

## Open Questions

1. **HRTB on `scope`**: The `for<'scope> AsyncFnOnce(&'scope TaskSpawner<'scope, E>)` bound may be hard to compile today. We may need to simplify — e.g., just parameterize over `'env` without the HRTB, or use a simpler signature and rely on the compiler inferring the lifetime correctly. To be determined during implementation.

2. **`MatchDispatch` chaining with `&self`**: Sequential `.if_request().await` calls should work with `&self` (shared ref released between each await). If we need `&mut self`, we'd have to restructure. Keep handlers on `&self` — use interior mutability or message-passing for mutations.

3. **Error propagation from spawned tasks**: A task returning `Err` propagates and cancels the scope. Recoverable errors (agent won't start) should be caught internally by the task, which responds to the client with an error and returns `Ok(())`. Only truly fatal errors propagate.

4. **Post-handshake dispatcher-to-agent requests**: Once the `AgentForwarder` is installed (claiming all dispatches), `block_task()` on that connection will hang — responses are intercepted. The dispatcher must never send requests to an agent after forwarder installation. All post-handshake traffic is pure forwarding. If we ever need dispatcher-initiated requests, we'd need a correlation mechanism.

## Implementation Notes

- `with_spawned_task` in the plan → actual API is **`with_spawned`** with signature `FnOnce(ConnectionTo<...>) -> impl Future`.
- `on_receive_dispatch` requires a second argument: `agent_client_protocol::on_receive_dispatch!()` (macro hack for async closure limitations).
- Agent pipes are `tokio::spawn`'d — they are `'static` tasks that communicate only via owned channel clones. They are NOT scoped tasks. Only short-lived work (like "wait for agent ready, then respond") goes into the scope's `TaskSpawner`.

## Deviations from Plan (recorded during implementation)

### Major deviation 1: `scope-tasks` not used in the dispatcher

**Root cause**: `Responder<T>` is `Send + !Sync`. Any struct containing responders (like the dispatcher's `pending_responses` map) makes an async future `!Send` when held across `.await` points. Since `tokio::spawn` requires `Send` futures, the dispatcher cannot use `scope_tasks::scope()` (which requires the caller's future to be spawnable).

**What the plan said**: Spawned tasks borrow non-`'static` data (like `Responder`s) from the scope, and the `TaskSpawner` handles structured teardown.

**What we did**: The dispatcher uses `tokio::spawn` directly for all background work (agent pipes, timer tasks, Initialize probes). All spawned work is inherently `'static` since it communicates via owned channel clones. The `scope-tasks` crate is built and tested but remains unused in the daemon — available as a standalone utility.

### Major deviation 2: Pending-response map instead of "responder captured in spawned task"

**Root cause**: Agent pipes run in independent `tokio::spawn` tasks. They can't hold the `Responder` directly because (a) they don't know the session_id until *after* the agent responds to session/new, and (b) the responder can't easily be threaded through the agent pipe API without making it generic.

**What the plan said**: The spawned task captures the `Responder`, awaits the agent pipe's oneshot signal, then responds.

**What we did**: The dispatcher stores `PendingResponse` in a `HashMap<AgentId, PendingResponse>` before spawning the agent pipe. When `AgentReady` arrives through the message channel, the dispatcher looks up and fulfills the pending responder. On `AgentSpawnFailed`, it responds with an error. Same FIFO ordering guarantee (AgentReady is enqueued before any client prompt can arrive).

### Major deviation 3: Manual method-string dispatch instead of `MatchDispatch`

**Root cause**: Same `!Send` issue. `MatchDispatch` chains `.if_request(async |req, responder| { self.handle_xxx(...) }).await` which holds `&mut self` across await points. With `!Sync` data in `self`, the future becomes `!Send`.

**What the plan said**: Use `MatchDispatch::new(dispatch).if_request(...).await.if_request(...).await.otherwise(...)`.

**What we did**: `match dispatch.method()` + `serde_json::from_value::<T>(req.params)` + `responder.cast::<T>()`. The dispatcher's `handle_message` and `handle_from_client` are fully synchronous — no `.await` anywhere except the top-level `rx.recv().await`. This eliminates all `Send`/`Sync` concerns.

### Minor deviations

- **HRTB on `scope` (Open Question 1)**: The `for<'scope> AsyncFnOnce(&'scope TaskSpawner<'scope, E>)` bound does not compile. Fixed by passing `TaskSpawner` by value: `impl AsyncFnOnce(TaskSpawner<'env, E>) -> ... + 'env`. Manual `Clone` impl avoids `E: Clone` bound.
- **Initialize probe spawned, not inline**: The plan said "blocks the dispatcher briefly." Since the dispatcher is fully synchronous, the probe runs in a background `tokio::spawn` that responds directly and sends `AgentCapabilities` back for caching.
- **`Dispatch` is not `Clone`**: Multi-client notification broadcast simplified to route to the most-recently-connected client only. In practice there's one client per session; multi-client broadcast would need JSON re-serialization.
- **No `AgentSpawnMode::Probe`**: The capabilities probe doesn't use `agent_pipe` at all — it's a one-shot `Client.builder().connect_with(...)` in a spawned task. `agent_pipe` only handles session establishment.
- **`on_receive_dispatch` in client pipe**: Works as designed. The `Dispatch::Request` variant carries the `Responder` through the channel — this is the key insight that makes the whole architecture work without typed handlers on the client side.
